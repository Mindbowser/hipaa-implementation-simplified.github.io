Developer Checklist: Part 1 (Detailed)

1

Understand HIPAA Regulations

Developers must be familiar with Privacy Rules, Security Rules, and Breach Notification Rules. This understanding guides all development decisions.

Example: When building a health app, knowing that birth dates are PHI means you'll store them encrypted and limit access through role-based controls.
2

Implement Access Controls

Restrict PHI access through authentication (passwords, MFA, biometrics) and role-based access control (RBAC).

Example: A nurse can only view records for patients in their care unit. A doctor has broader access. A billing clerk sees only financial data, not clinical notes.
3

Data Encryption

Transform data into coded form that requires a key to decrypt. Essential for protecting PHI both at rest (stored) and in transit (transmitted).

Example: Patient records in your database are encrypted with AES-256. If a hacker gains database access, they see only ciphertext without the encryption keys.
4

Audit Trails

Track and record all PHI access and modifications. Logs must be immutable and include: who, what, when, and why.

Example: Every time a user opens a patient file, log: timestamp, user ID, action type, patient ID, and access reason. Store logs securely for 6+ years.

Developer Checklist: Part 2 (Detailed)

5

Secure Development Practices

Write code that minimizes vulnerabilities. Perform regular security testing including SAST, DAST, and penetration testing.

Example: Use parameterized queries to prevent SQL injection. Implement Content Security Policy headers. Run OWASP ZAP scans before each release.
6

Data Minimization

Collect and retain only the minimum necessary PHI for your intended purpose. Less data = less risk.

Example: If verifying age for prescription eligibility, ask for birth year only ‚Äî not full birthdate, address, and SSN.
7

Secure Transmission of Data

Encrypt all data transmitted over networks using TLS 1.2+ (HTTPS), VPNs, or secure file transfer protocols.

Example: Mobile app submits patient vitals to your API. Use HTTPS with certificate pinning to prevent man-in-the-middle attacks.
8

Business Associate Agreements (BAAs)

Contracts with vendors that handle PHI ensuring they'll safeguard it appropriately. Required by law.

Example: Using AWS S3 for patient document storage? Sign AWS's BAA. Using Zoom for telehealth? Sign Zoom's BAA. No BAA = HIPAA violation.

Developer Checklist: Part 3 (Detailed)

9

Disaster Recovery & Contingency Planning

Maintain plans for data backup, disaster recovery, and emergency operations to ensure PHI can be restored.

Example: Automated daily backups to geographically separate location. Quarterly disaster recovery drills. Recovery Time Objective (RTO) of 4 hours.
10

Employee Training & Awareness

Regular training for all employees who handle PHI. Document training completion and update annually.

Example: Annual HIPAA training covering: recognizing PHI, secure communication, incident reporting, and penalties. New hires complete training before PHI access.
11

Regular Security Assessments

Conduct periodic security audits and vulnerability assessments to identify and fix security issues proactively.

Example: Quarterly vulnerability scans. Annual penetration testing by third-party firm. Monthly dependency checks for vulnerable libraries.
12

Incident Response Plan

Documented plan for identifying, responding to, and mitigating security incidents involving PHI.

Example: 1) Detect breach via monitoring. 2) Contain: isolate affected systems. 3) Investigate: determine scope. 4) Notify: individuals within 60 days. 5) Remediate: fix vulnerability.

Access Controls & Data Minimization

Role-Based Access Control (RBAC) + Data Minimization

RBAC assigns permissions based on user roles and enforces the principle of least privilege. Combined with data minimization, each role only accesses the minimum necessary PHI for their specific tasks.

Common Healthcare Roles

Patient

View own records, request appointments, message providers

Nurse

View assigned patients (limited fields), update vitals, add notes

Doctor

Full access to patient records, prescribe medications, order tests

Admin

User management, system configuration, audit logs

Implementation Principles

  • Least Privilege: Grant minimum access needed
  • Data Minimization: Return only necessary fields per role
  • Separation of Duties: No single user has complete control
  • Need-to-Know: Access only when clinically necessary
  • Time-Limited: Temporary elevated permissions expire
  • Audit Trail: Log all access attempts and changes

üí° Data Minimization in Practice

Example: A nurse checking patient age for medication dosage doesn't need the full date of birth (1985-03-22). Return only the birth year (1985) or calculated age (39 years). This reduces exposure if the session is compromised.

Doctor viewing full medical history? Full DOB provided. Billing clerk processing invoice? Only age range needed, no DOB at all.

Implementation Example (Node.js)

// Middleware for role-based access with data minimization
function requireRole(...allowedRoles) {
  return (req, res, next) => {
    if (!req.user) {
      return res.status(401).json({ error: 'Not authenticated' });
    }
    
    if (!allowedRoles.includes(req.user.role)) {
      // Log failed access attempt
      auditLog.record({
        user: req.user.id,
        action: 'ACCESS_DENIED',
        resource: req.path,
        timestamp: new Date()
      });
      return res.status(403).json({ error: 'Insufficient permissions' });
    }
    
    // Apply data minimization based on role
    req.dataFields = getMinimumFields(req.user.role);
    next();
  };
}

function getMinimumFields(role) {
  const fieldMap = {
    'nurse': ['name', 'birthYear', 'age', 'mrn', 'diagnosis'],  // No full DOB
    'doctor': ['name', 'dob', 'mrn', 'diagnosis', 'medications', 'allergies'],
    'admin': ['userId', 'role', 'lastLogin']
  };
  return fieldMap[role] || [];
}

// Protected routes
app.get('/api/patients/:id', 
  requireRole('doctor', 'nurse'), 
  patientController.getRecord);  // Returns only fields in req.dataFields

Data Encryption: At Rest & In Transit

Encryption at Rest

Protects stored data on disks, databases, and backups

  • AES-256 for symmetric encryption
  • Transparent Data Encryption (TDE)
  • Full-disk encryption
  • Application-level encryption

Encryption in Transit

Protects data moving across networks

  • TLS 1.2+ for HTTPS
  • Certificate pinning
  • VPN tunnels
  • SFTP for file transfers

Encryption Workflow

Encryption Best Practices

Key Management

Use dedicated key management services (AWS KMS, Azure Key Vault). Rotate keys regularly.

Algorithm Choice

Use industry-standard algorithms: AES-256, RSA-2048+, SHA-256 or better.

Never Roll Your Own

Use vetted libraries (OpenSSL, Crypto APIs). Custom encryption is vulnerable.

Defense in Depth

Layer encryption with access controls, network segmentation, and monitoring.

Why This Matters

If a database backup is stolen or a laptop is lost, encrypted data remains protected. Without the encryption key (stored separately in a Key Management Service), the attacker sees only gibberish.

Real Example

A healthcare provider's unencrypted backup tapes were stolen from a courier's car. 4.9 million patient records were exposed. Settlement: $5.5 million.

If those tapes had been encrypted, no breach notification would have been required.

Audit Trails: Tracking Every Access

What to Log

Every access or modification of PHI must be logged with immutable records.

Who

User ID, name, role, IP address, session ID

What

Action performed (READ, WRITE, UPDATE, DELETE)

When

Timestamp (with timezone), duration

Where

Resource accessed (patient ID, record type)

Audit Log Requirements

  • Immutable: Logs cannot be edited or deleted by regular users
  • Retention: Store for minimum 6 years (check state laws)
  • Secure: Encrypt logs and restrict access to administrators
  • Monitoring: Alert on suspicious patterns (bulk downloads, after-hours access)
  • Regular Review: Periodic audits to detect policy violations

Business Associate Agreements (BAAs)

What is a BAA?

A legally binding contract between a HIPAA covered entity and a vendor (business associate) that handles PHI. The BAA ensures the vendor will properly safeguard PHI.

Common Business Associates

Cloud Providers

AWS, Azure, Google Cloud (when storing PHI)

SaaS Tools

CRM, analytics, email marketing with PHI

Telehealth

Zoom, Vonage, Twilio for patient communication

Billing Services

Third-party medical billing companies

BAA Must Include

  • Permitted uses of PHI
  • Safeguard requirements
  • Restriction on further disclosure
  • Subcontractor requirements
  • Breach notification obligations
  • Return/destruction of PHI at termination
  • Right to audit compliance

‚ö†Ô∏è Warning: No BAA = HIPAA Violation

Using a vendor without a BAA when they handle PHI is itself a HIPAA violation, even if no breach occurs.

Example: A clinic used Google Analytics on their patient portal without a BAA. They were fined even though no breach happened, because sharing PHI with Google without a BAA violated HIPAA.

‚ö†Ô∏è Common AI Compliance Mistakes:

  • Using consumer AI tools with PHI: ChatGPT web interface, free Claude - NO BAA available, don't use with PHI
  • Sending PHI to AI APIs without BAAs: Even if using API (not web), you need a signed BAA
  • Training models on PHI: If vendor trains their models on your data, they need BAA + must commit to not using PHI for training
  • Third-party AI integrations: Embedded AI widgets, chatbots, or tools on your site need BAAs

‚úÖ AI BAA Best Practices:

  • Use enterprise/API versions: Azure OpenAI, AWS Bedrock offer BAAs (consumer versions don't)
  • Verify no training on your data: Ensure vendor doesn't use your PHI to improve their models
  • Data retention policies: How long does AI vendor keep your data? Should be minimal or zero
  • De-identify before sending: Remove/mask PHI before sending to AI if BAA unavailable
  • Audit AI access: Log what data is sent to AI services, track all API calls

‚ùå NEVER paste PHI into ChatGPT, Claude web interface, or any consumer AI tool without a BAA. Use only enterprise API versions with signed BAAs.

Incident Response Plan

Preparing for the Inevitable

Not if, but when. Having a documented incident response plan is crucial.

Breach Notification Timeline

  • Discovery: Incident identified through monitoring or report
  • Day 1-10: Contain breach, assess scope, preserve evidence
  • Day 11-30: Complete investigation, determine affected individuals
  • Day 31-60: Notify affected individuals (60-day deadline)
  • Concurrent: Report to HHS (within 60 days)
  • If 500+: Notify media and HHS immediately

Notification Requirements

Breaches affecting 500 or more individuals require notification to prominent media outlets and are posted on HHS's "Wall of Shame" public website.

Smaller breaches must still be reported to HHS within 60 days of discovery.

Common HIPAA Mistakes to Avoid

Learn from Others' Errors

These are the most frequent compliance mistakes that lead to breaches and fines.

‚ùå Assuming Cloud = Compliant

Just because you use AWS or Azure doesn't mean you're HIPAA compliant. You must configure services correctly and sign a BAA.

‚ùå Logging PHI in Plain Text

Application logs, error messages, and debug output often contain PHI. Use tokenization or redact sensitive data.

‚ùå Using Tools Without BAA

Google Analytics, Intercom, Mixpanel on patient-facing sites without BAAs violates HIPAA.

‚ùå Weak Access Controls

Shared passwords, no MFA, excessive permissions, no session timeouts.

‚ùå Unencrypted Transmission

Sending PHI via regular email, SMS, or HTTP instead of encrypted channels.

‚ùå No Audit Logs

Not logging access to PHI or making logs editable by users.

‚ùå Ignoring Updates

Running outdated software with known vulnerabilities (patch management failure).

‚ùå No Employee Training

Assuming developers know HIPAA without formal training.

üîë How to Retrieve Data from Debug Token

Debug tokens are one-way hashes - you cannot directly reverse them. Store a secure mapping to retrieve patient IDs when debugging:

Store Mapping in Secure Location

// Store mapping in encrypted database (separate from logs)
const tokenMapping = {
  'a7f3c2b89e4d': 'pat_002',
  'b8e4d3c7a9f1': 'pat_005'
};

// Retrieve when needed
function lookupPatientId(debugToken) {
  return tokenMapping[debugToken];
}

// Usage during debugging
const patientId = lookupPatientId('a7f3c2b89e4d');
console.log(patientId); // 'pat_002'

üí° Best Practice: Store the token-to-ID mapping in a separate encrypted database table with strict access controls. Only security/ops teams should have access. This way logs remain PHI-free, but you can still debug production issues when needed.

‚ö†Ô∏è Security Note: The SECRET key in process.env.SECRET must be kept secure and rotated regularly. Without it, tokens cannot be regenerated to match against known IDs.
